# Kapable Rust API Platform ŌĆö Comprehensive Knowledge Document\n\n## 1. Overview\n\n**What:** The Kapable Rust API is the core backend platform for Kapable, an AI-native Backend-as-a-Service (BaaS) system. It provides REST APIs for data management, authentication, application deployment, serverless functions, AI flows, and platform services (email, storage, video, voice).\n\n**Scope:** Multi-tenant Rust backend using Axum (HTTP), SQLx (async PostgreSQL), Tokio (async runtime), and tokio-stream (SSE/WebSocket support). Compiled to 6 binaries + 1 library + 1 migration runner.\n\n**Fit in Platform:** Central API serving three frontend consoles (admin, developer, organization), internal services (worker scheduler, proxy gateway, AI daemon), and external SDK clients. Single entry point for all platform operations.\n\n**Not In Production:** Can change architecture freely. No migration burden.\n\n---\n\n## 2. Authentication System\n\n### Auth Tiers\n\nEvery endpoint declares its auth tier. API enforces the declared tier via middleware:\n\n| Tier | Credential | Scope | Use Case | Middleware |\n|------|-----------|-------|----------|-----------|\n| `admin_only` | `sk_admin_*` API key OR admin session | Platform-wide operations | Create org, manage billing, view all apps | `require_admin_key_or_session()` |\n| `org_member` | Session token (`kses_*`) | Single org | Access org dashboard, settings, members | `require_session()` + `require_org_context()` |\n| `data_key` | API key (`sk_live_*` or `sk_test_*`) | Project-scoped, may have scopes (e.g., `read:users`) | Client SDKs, data CRUD | `validate_api_key()` + `require_scope()` (optional) |\n| `token_bearer` | JWT data token (end-user issued by BetterAuth) | Project-scoped, user-scoped | Frontend, post-login | `verify_token()` |\n| `public` | None | None | Health checks, OpenAPI schema | No auth middleware |\n\n### No `users` Table\n\n**CRITICAL:** The platform uses org-centric identity. NO `users` table exists.\n\n| Table | Key Columns | Purpose |\n|-------|-------------|---------|\n| `organizations` | `id`, `slug`, `name`, `plan` | Tenant record |\n| `org_members` | `id`, `org_id`, `email`, `password_hash`, `role`, `name` | **User identity ŌĆö email stored here directly** |\n| `sessions` | `id`, `user_id` (ŌåÆ org_members.id), `user_type`, `token`, `expires_at` | Login session (kses_*) |\n| `api_keys` | `id`, `org_id`, `project_id`, `key_prefix`, `key_hash`, `scopes` | API key (sk_live_*, sk_test_*, sk_admin_*) |\n| `password_reset_tokens` | `id`, `user_id` (ŌåÆ org_members.id), `token`, `expires_at` | Password reset flow |\n| `service_tokens` | `id`, `org_id`, `key_type`, `key_hash` | Service tokens (st_ci_*, st_wh_*) |\n| `sso_providers` | `id`, `org_id`, `provider_type`, `domain` | SSO configuration |\n| `sso_sessions` | `id`, `org_id`, `provider_id`, `user_email` | SSO login sessions |\n\n### Per-Project Auth (BetterAuth)\n\nEach project schema has its own auth table:\n- Table: `{schema}._auth_users` (created per project, NOT on public schema)\n- Used for: End-user signup/login via `/v1/auth/` endpoints in Connect apps\n- JWT issued: HS256, per-project secret, scoped to `{project_id}:{user_id}`\n\n### Auth Context Types\n\nUnified auth context (Phase D) ŌĆö routes match on the variant they need:\n\n```rust\npub enum AuthContext {\n    PlatformAdmin(AdminContext),       // sk_admin_* key or admin session\n    OrgMember(SessionContext),         // org member session\n    ApiKey(ApiContext),                // sk_live_* or sk_test_*\n    JwtToken(JwtTokenContext),         // end-user JWT\n    ServiceToken(ServiceTokenContext), // st_ci_* or st_wh_*\n    Internal(AdminContext),            // service-to-service\n}\n```\n\n### Session Model\n\n- Token format: `kses_` + 59 random characters\n- Expires: 30 days (configurable)\n- Per-instance: No cross-instance session invalidation (each API instance validates independently)\n- Destruction: Sessions remain in DB; app checks `expires_at > NOW()` on every request\n\n### Common Auth Patterns\n\n**Org member (session-based) ŌåÆ org_members:**\n```sql\nSELECT om.id, om.email, om.role, o.id as org_id\nFROM sessions s\nJOIN org_members om ON om.id = s.user_id\nJOIN organizations o ON o.id = om.org_id\nWHERE s.token = $1 AND s.expires_at > NOW()\n```\n\n**API key ŌåÆ api_keys ŌåÆ org:**\n```sql\nSELECT ak.id, ak.org_id, ak.project_id, ak.scopes,\n       p.schema_name, ak.tier\nFROM api_keys ak\nLEFT JOIN projects p ON p.id = ak.project_id\nWHERE ak.key_hash = $1 AND ak.revoked_at IS NULL\n```\n\n**JWT token:**\n- Decoded via `jsonwebtoken::decode(token, key, &validation)`\n- `sub` field contains `{project_id}:{user_id}`\n- Scopes: optional `scopes` claim (object or array)\n\n---\n\n## 5. Data API\n\n### How It Works\n\nDynamic, schema-agnostic CRUD for project tables. Supports:\n- **JSONB mode** (schemaless) ŌĆö store arbitrary JSON in `data` column\n- **Typed mode** ŌĆö strongly-typed columns (TEXT, UUID, TIMESTAMPTZ, NUMERIC, JSONB, BOOLEAN, TEXT[], UUID[])\n- **Real-time SSE** ŌĆö WAL-based change stream via `/v1/sse`\n- **JWT scoping** ŌĆö scope filters applied automatically to queries\n- **RLS** (Row-Level Security) ŌĆö per-org data isolation via PostgreSQL policies\n\n### Key Routes\n\n**List/Create:**\n- `GET /v1/{project_id}/{table}?limit=50&offset=0&order_by=created_at&order=asc`\n- `POST /v1/{project_id}/{table}` ŌĆö create row\n\n**Read/Update/Delete:**\n- `GET /v1/{project_id}/{table}/{id}`\n- `PATCH /v1/{project_id}/{table}/{id}` ŌĆö partial update (COALESCE pattern)\n- `DELETE /v1/{project_id}/{table}/{id}`\n\n**Metadata:**\n- `GET /v1/_meta/tables` ŌĆö list all tables in project\n- `POST /v1/_meta/tables` ŌĆö create table (schema DDL)\n- `GET /v1/_meta/tables/{table}` ŌĆö get table schema\n- `PUT /v1/_meta/tables/{table}` ŌĆö update table schema\n- `DELETE /v1/_meta/tables/{table}` ŌĆö drop table\n\n**Scoping (JWT):**\n- Scope: `read:users`, `write:users`, `delete:users`\n- Route prefix: `/v1/scope/{project_id}/{table}`\n- Same endpoints; scope filters injected via `generate_scope_filters()`\n\n### Auth for Data API\n\nThree paths (in order of precedence):\n\n1. **API key** (`sk_live_*` or `sk_test_*`) ŌĆö full access, all scopes\n2. **JWT token** (data token from BetterAuth) ŌĆö scoped access, filters injected\n3. **Session + X-Project-Id** (BFF proxy) ŌĆö org member session + project header\n\nHeader extraction:\n- Auth: `Authorization: Bearer <key>` or `X-API-Key: <key>`\n- For session: `X-Session-Token: <token>` + `X-Project-Id: <uuid>`\n\n### Real-Time (SSE)\n\n**Endpoint:** `GET /v1/sse?project_id={id}&branch={optional}`\n\n**Flow:**\n1. WAL consumer listens to `kapable_platform_changes` publication\n2. Decodes logical replication messages\n3. BroadcastManager fans out to per-project channels\n4. SSE endpoint subscribes and streams events\n\n**Event types:**\n- `INSERT` ŌĆö new row in table\n- `UPDATE` ŌĆö modified row\n- `DELETE` ŌĆö deleted row\n\n**Client-side:** Frontend should use exponential backoff (min 2s, max 30s) for reconnection to avoid feedback loops with `last_used_at` updates.\n\n### Filters\n\nQuery params: `?filter=column:eq:value&filter=age:gt:18`\n\nSupported operators: `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, `like`, `in`, `contains` (JSONB)\n\n### Branch Support\n\nQuery param: `?branch=feature-branch`\n\nRoutes request to `{schema}_{branch}` (e.g., `public_feature-branch` for schema)\n\n---\n\n## 6. Gotchas\n\n### No `users` Table\n\nEVERY query references `org_members` directly, never `users`. Common mistake:\n\n```rust\n// WRONG\nsqlx::query(\"SELECT email FROM users WHERE id = $1\")\n\n// RIGHT\nsqlx::query(\"SELECT email FROM org_members WHERE id = $1\")\n```\n\n### Specific Routes Before Wildcards\n\nRouter matching order matters. Bad:\n\n```rust\nRouter::new()\n    .route(\"/v1/:table\", get(list_table))         // matches /v1/users, /v1/_meta\n    .route(\"/v1/_meta/tables\", get(meta_tables))  // NEVER HIT\n```\n\nGood:\n\n```rust\nRouter::new()\n    .route(\"/v1/_meta/tables\", get(meta_tables))\n    .route(\"/v1/:table\", get(list_table))\n```\n\n### COALESCE in UPDATE Queries\n\nNULL optional fields must NOT overwrite existing values:\n\n```rust\n// WRONG ŌĆö NULL overwrites existing name\nsqlx::query(\n    \"UPDATE users SET email = $1, name = $2 WHERE id = $3\"\n)\n    .bind(new_email)\n    .bind(new_name_or_none)  // ŌåÉ WRONG if None\n    .bind(user_id)\n\n// RIGHT ŌĆö use COALESCE\nsqlx::query(\n    \"UPDATE users SET email = $1, name = COALESCE($2, name) WHERE id = $3\"\n)\n    .bind(new_email)\n    .bind(new_name_or_none)\n    .bind(user_id)\n```\n\n### DashMap Deadlock Pattern\n\nNEVER bind `.get()` result to a named variable if next code touches the same key:\n\n```rust\n// WRONG ŌĆö deadlock risk on same-shard concurrent access\nlet guard = event_sinks.get(run_id);\nevent_sinks.insert(run_id, new_sink);  // ŌåÉ DEADLOCK\n\n// RIGHT ŌĆö drop guard before second operation\n{\n    let old = event_sinks.get(run_id);\n    drop(old);  // explicit\n}\nevent_sinks.insert(run_id, new_sink);\n\n// OR ŌĆö single expression\nevent_sinks.remove(run_id).map(|(_, v)| v)\n```\n\n### RLS Not Applied by Default\n\nRow-level security in PostgreSQL requires explicit role setup. If querying as superuser (app does), RLS policies are BYPASSED. Must set proper role or use `current_setting()` gating:\n\n```sql\nSET ROLE {org_schema}_role;  -- Before querying\n```\n\nOr rely on app-level filtering (preferred, simpler).\n\n### No Outer Scope Variables in PL/pgSQL\n\nInside EXECUTE format() in PL/pgSQL functions, variables must be passed explicitly:\n\n```sql\n-- WRONG\nCREATE FUNCTION my_func(org_id UUID) AS $$\nDECLARE\n    result JSONB;\nBEGIN\n    EXECUTE format('SELECT * FROM %I WHERE org_id = %L', table_name, org_id);\n    -- Ōåæ org_id is IN SCOPE but EXECUTE string doesn't see it\nEND;\n$$ LANGUAGE plpgsql;\n\n-- RIGHT\nCREATE FUNCTION my_func(org_id UUID) AS $$\nDECLARE\n    result JSONB;\nBEGIN\n    EXECUTE format('SELECT * FROM %I WHERE org_id = %L', table_name, org_id)\n        INTO result;\n    -- Ōåæ INTO captures result\nEND;\n$$ LANGUAGE plpgsql;\n```\n\n### SSE Exponential Backoff\n\nBrowser auto-reconnects EventSource on close with ~3s fixed interval. Combined with frequent `last_used_at` DB updates, causes feedback loop:\n\nBad: SSE closes ŌåÆ browser reconnects 3s later ŌåÆ API updates `last_used_at` ŌåÆ next request ŌåÆ 429 rate limit\n\nGood: Use exponential backoff on client (min 2s, max 30s, double on error)\n\n### Cargo Process Discipline\n\nNever spawn two cargo processes in the same workspace simultaneously. They fight for build lock:\n\n```bash\n# Session A\ncargo build  # acquires lock\n\n# Session B\ncargo test   # waits for lock, never gets it, hangs forever\n```\n\nCheck `ps aux | grep cargo` before any build command.\n\n### ForeignKey Constraints in Migrations\n\nEvery FK needs explicit `ON DELETE` behavior. Missing it causes implicit CASCADE (dangerous):\n\n```sql\n-- WRONG ŌĆö implicit CASCADE on delete\nALTER TABLE pipelines\n  ADD CONSTRAINT fk_pipeline_org\n  FOREIGN KEY (org_id) REFERENCES organizations(id);\n\n-- RIGHT ŌĆö explicit behavior\nALTER TABLE pipelines\n  ADD CONSTRAINT fk_pipeline_org\n  FOREIGN KEY (org_id) REFERENCES organizations(id)\n  ON DELETE CASCADE\n  ON UPDATE CASCADE;\n```\n\n### Metrics Labels\n\nPrometheus doesn't support arbitrary label values (especially UUIDs). Normalize paths:\n\n```rust\n// WRONG\nmetrics::counter!(\"api_calls\", \"org_id\" => org_id.to_string()).increment(1);\n// Ōåæ Creates 10000s of series for 10000 orgs\n\n// RIGHT\nmetrics::counter!(\"api_calls\", \"endpoint\" => \"/v1/data/{table}\").increment(1);\n```\n\n---\n\n## 14.
