# ## Koncierge Research Report: AI-Powered Onboarding Assistants\n\n### Top 5 Products/Projects to Study\n\n**1. CopilotKit (open source, github.com/CopilotKit/CopilotKit)**\n\nThe single most relevant project. It is an open-source React framework purpose-built for embedding AI copilots in apps. Key features that map directly to Koncierge requirements:\n- `useFrontendTool()` hook lets the AI agent trigger app-side actions (navigation, highlighting)\n- `useAgentContext()` provides bidirectional state sync -- the agent knows what page the user is on, and can mutate app state\n- AG-UI Protocol: 16 real-time event types over SSE connecting agent to frontend\n- Generative UI: agent can dynamically render React components inline in chat\n- Human-in-the-loop: agent pauses for user confirmation\n- Supports Claude/Anthropic via the Vercel AI SDK adapter\n\n**Why study it:** The `useFrontendTool` + `useAgentContext` pattern is exactly what Koncierge needs for route-aware context injection and navigate/highlight actions. You could potentially use CopilotKit directly, or steal the architectural pattern.\n\n**2. assistant-ui (github.com/assistant-ui/assistant-ui, YC-backed)**\n\nThe best open-source React chat UI component library. It gives you the chat panel itself:\n- Composable primitives built on Radix UI (same foundation as your shadcn components)\n- Built-in streaming (25ms throttled rendering), auto-scroll, markdown, code highlighting\n- Voice input (dictation) out of the box\n- Message branching, retry, edit\n- Works with Anthropic Claude directly via Vercel AI SDK or custom adapters\n- AssistantStreamChunk protocol with TextDelta, ToolCallBegin, ToolResult events\n\n**Why study it:** If you build Koncierge's chat panel from scratch, use this as the UI layer. It handles all the SSE rendering, auto-scroll, and accessibility that would take weeks to get right.\n\n**3. Command AI (formerly CommandBar, command.ai)**\n\nThe commercial product closest to Koncierge's vision. It combines:\n- AI chat agent embedded in-app (not a separate window)\n- \"Co-browse\" mode: visually guides users through the UI with highlighting\n- API action ingestion: agent can execute backend actions on behalf of the user\n- Proactive nudges: segment-based contextual suggestions\n- Knowledge base training from help docs, resolved conversations, internal content\n- Human handoff when AI is stuck\n\n**Why study it:** Their \"co-browse\" feature (AI triggers in-app walkthroughs dynamically) is exactly what Koncierge Phase 2 needs. The pattern of \"chat + highlight + navigate\" in one product is validated here.\n\n**4. ElevenLabs Conversational AI Widget (@elevenlabs/react)**\n\nFor Phase 3 (voice), this is the most production-ready embeddable voice agent:\n- `useConversation` React hook manages WebSocket/WebRTC connections and audio\n- Sub-100ms latency with Flash v2.5 model\n- Dual mode: voice + text, users can switch mid-conversation\n- Shadow DOM encapsulation for clean embedding\n- Transcript pipeline with streaming text accumulation\n- 10-minute idle timeout for text-only sessions\n- You already have `ELEVENLABS_API_KEY` configured on production\n\n**Why study it:** When you add voice, you can use `@elevenlabs/react` directly rather than building a voice pipeline from scratch.\n\n**5. Intercom Fin (fin.ai)**\n\nThe most sophisticated commercial AI agent for customer support. Architecture worth borrowing:\n- Three-layer RAG pipeline: Query Refinement -> Retrieval -> Reranking -> Generation\n- Custom `fin-cx-retrieval` and `fin-cx-reranker` models for domain-specific accuracy\n- 99.9% accuracy rate by grounding in multiple knowledge sources\n- \"Procedures\" system: multi-step resolution workflows (not just Q&A)\n- Content gap detection: automatically identifies missing knowledge\n- Channel-aware responses: adjusts format based on context (email vs chat vs in-app)\n\n**Why study it:** The query refinement + reranking pattern is worth adopting. Even with a full-context approach (not RAG), you can apply query refinement to determine which section of KAPABLE_KNOWLEDGE_BASE.md is most relevant to the user's current page + question.\n\n---\n\n### Key Patterns to Adopt\n\n**1. Route Context as First-Class Input**\n\nEvery message to the AI should include: current route, visible data entities (org_id, app_id, etc.), and user role. CopilotKit does this with `useAgentContext()` that pushes reactive state to the agent. For Koncierge, this means wrapping every user message with:\n```\n{ route: \"/apps/abc/environments\", entities: { app_id: \"abc\" }, role: \"admin\", message: \"How do I deploy?\" }\n```\n\n**2. Tool-Based Actions (Not String Parsing)**\n\nDefine structured tools the agent can call: `navigate({ path })`, `highlight({ selector, tooltip })`, `showChecklist({ items })`, `openSection({ name })`. CopilotKit's `useFrontendTool()` and Command AI's action API both use this pattern. The agent returns tool calls in the response stream; the frontend executes them.\n\n**3. Prompt Caching for Warm Sessions**\n\nAnthropic's prompt caching is the mechanism for \"warm sessions.\" Cache the system prompt (KAPABLE_KNOWLEDGE_BASE.md) with `cache_control`, and subsequent messages within 5 minutes reuse the cached prefix. This gives 85% latency reduction and 90% cost reduction for the knowledge base portion. No need for a separate \"warm session\" daemon -- the API's cache is the warm pool.\n\n**4. Driver.js for Programmatic Highlighting**\n\nAt ~5KB gzipped, driver.js is the lightest option for element highlighting/annotation. Its API allows programmatic `highlight({ element, popover })` calls -- perfect for the AI to trigger via tool calls. No state machine needed (unlike react-joyride), just imperative `driverInstance.highlight(selector)` when the agent says to.\n\n**5. Streaming via SSE with Structured Events**\n\nUse SSE (not WebSocket) for the chat response stream. The Anthropic API already streams via SSE. The assistant-ui AssistantStreamChunk protocol is a good event model: `TextDelta`, `ToolCallBegin`, `ToolCallDelta`, `ToolResult`. The frontend can render text tokens as they arrive AND execute highlight/navigate tools as they appear in the stream.\n\n**6. Dual-Mode Voice (Push-to-Talk Default)**\n\nVoice UX research shows push-to-talk is more reliable than auto-detect, especially in browser. The Web Speech API (Chrome/Edge only) is free but limited. For production: use push-to-talk with ElevenLabs for TTS (streaming, low latency) and Web Speech API for STT (free, adequate for Chrome). Fall back to text for unsupported browsers. Industry latency expectation: 600-800ms for response start.\n\n---\n\n### Key Anti-Patterns to Avoid\n\n**1. Don't Use RAG for a Known, Bounded Knowledge Base**\n\nWith ~150K tokens of platform knowledge, you are within Claude's context window. RAG adds retrieval latency, chunk boundary problems, and citation complexity. Full-context with prompt caching is simpler, faster, and more accurate for this use case. RAG is for unbounded/dynamic corpora (millions of docs). Your knowledge base is bounded and refreshed periodically.\n\n**2. Don't Auto-Trigger Tours/Highlights Without User Action**\n\nResearch from NN/G and Chameleon benchmarks shows: proactive guidance that interrupts the user's flow has 30%+ abandonment. Launcher-driven tours (user initiates) reach 67% completion vs ~20% for auto-triggered popups. Koncierge should respond to questions, not ambush with tooltips. Exception: a one-time \"welcome\" greeting on first login is acceptable.\n\n**3. Don't Use Continuous Listening for Voice**\n\nWeb Speech API continuous listening on Android Chrome \"constantly restarts the microphone, leading to a frustrating and noisy (beeping) experience.\" Push-to-talk with a mic button is the safe choice. Auto-detect VAD is only reliable with server-side processing (LiveKit/Pipecat), which adds infrastructure complexity.\n\n**4. Don't Build a Monolithic Chat Component**\n\nassistant-ui's composable primitive approach (ThreadPrimitive, MessagePrimitive, ComposerPrimitive) is proven better than monolithic chat widgets. It lets you customize every part independently and integrate tool-call rendering (highlight actions, navigation actions) without fighting a rigid component structure.\n\n**5. Don't Let the Agent Hallucinate About Platform State**\n\nGround every answer in the knowledge base. Use system prompt instructions: \"Only answer based on the provided KAPABLE_KNOWLEDGE_BASE. If the answer is not in the knowledge base, say you don't know and suggest the user check the docs or ask a teammate.\" Anthropic's 2025 research shows refusal can be trained as behavior -- configure the system prompt to prefer \"I don't know\" over fabrication.\n\n**6. Don't Poll for Real-Time Updates**\n\nPer your existing platform rules: SSE for everything. The chat response stream should be SSE. Session status updates should be SSE. Never `setInterval` polling for message status.\n\n---\n\n### Specific Libraries/Components Worth Using\n\n| Library | Purpose | Size | Notes |\n|---------|---------|------|-------|\n| **@assistant-ui/react** | Chat UI primitives | Modular (tree-shakeable) | Radix-based, shadcn-compatible, streaming built-in |\n| **driver.js** | Element highlighting/annotation | ~5KB gzipped | Programmatic API, no React dependency needed |\n| **@elevenlabs/react** | Voice TTS/STT | SDK package | `useConversation` hook, WebRTC, you have the API key |\n| **react-speech-recognition** | Browser STT (free) | Lightweight wrapper | Chrome/Edge only, push-to-talk pattern |\n| **@copilotkit/react-core** | Frontend tool pattern | If adopting the framework | `useFrontendTool` + `useAgentContext` pattern |\n| **Vercel AI SDK** (`ai` package) | Streaming protocol | Core package | `useChat` hook, SSE handling, Anthropic provider |\n\n**Recommended Stack for Koncierge:**\n- Chat UI: assistant-ui primitives (matches your shadcn/Radix stack)\n- Streaming: Vercel AI SDK or direct Anthropic SSE (you already have the streaming pattern from KAIT)\n- Highlighting: driver.js (imperative, called via agent tool responses)\n- Voice TTS: @elevenlabs/react (Phase 3)\n- Voice STT: react-speech-recognition or Web Speech API direct (Phase 3)\n- Agent backend: Direct Anthropic Messages API with prompt caching on the BFF\n\n---\n\n### Architecture Ideas You May Not Have Considered\n\n**1. OnboardJS State Machine for Multi-Step Flows**\n\nOnboardJS separates the onboarding state machine from the UI rendering. Koncierge could maintain a state machine for onboarding progress: `{ completed: [\"create-org\", \"invite-member\"], current: \"create-app\", remaining: [\"deploy\", \"connect-domain\"] }`. The AI consults this state to know what the user has/hasn't done, and the state persists across sessions via org_members metadata.\n\n**2. AG-UI Protocol for Agent-Frontend Communication**\n\nCopilotKit's AG-UI protocol (adopted by Google, LangChain, AWS, Microsoft) defines 16 event types for agent-to-UI communication. Rather than inventing your own tool-call-to-action mapping, consider adopting this protocol. It handles TextDelta, ToolCallStart, ToolCallArgs, StateSnapshot, and more. This would make Koncierge interoperable with any AG-UI-compatible agent framework.\n\n**3. MCP Server for Platform Context**\n\nInstead of a static knowledge base only, expose a Koncierge MCP server that can query live platform state: `list-apps`, `get-app-status`, `check-deploy-health`, `get-user-onboarding-progress`. The agent calls these tools to ground answers in real-time data, not just documentation. This extends the knowledge base with live telemetry without bloating the system prompt.\n\n**4. Conversation Memory via Prompt Cache Chains**\n\nAnthropic's prompt caching creates a natural \"warm session\" without custom infrastructure. Structure the prompt as: `[System: KAPABLE_KB (cached)] + [Conversation history (growing)] + [Current message with route context]`. The KB stays cached for 5 minutes (refreshed on each message), conversation history grows, and you get sub-second first-token latency because the KB prefix is already processed.\n\n**5. Embedded Resource Center Pattern (Pendo-Style)**\n\nBeyond chat, add a \"Resource Center\" component (a la Pendo) that shows contextual help cards based on the current route. These are not AI-generated -- they are pre-written cards mapped to routes. The AI chat is for dynamic questions; the resource center is for discoverable, predictable help. This dual approach (static cards + dynamic AI) covers both browsing users and searching users.\n\n**6. Chameleon's \"Copilot Describes, System Builds\" Pattern**\n\nChameleon's AI copilot takes a natural language description (\"guide users through creating their first app\") and automatically generates the tour steps, targeting rules, and copy. Apply this to Koncierge: an admin describes an onboarding goal, and Koncierge generates the driver.js tour definition automatically, stored as a reusable template.\n\n---\n\n### Answering the Open Questions from PLAN.md\n\n**WebSocket vs SSE for chat streaming?**\nSSE. Anthropic's API streams via SSE natively. WebSocket adds bidirectional complexity you don't need (user messages are simple POSTs). SSE also works through your existing Caddy reverse proxy without special configuration.\n\n**Should Koncierge have write access or read-only?**\nStart read-only + navigate. The \"navigate to page\" and \"highlight element\" actions are client-side only (no API writes). Later, consider MCP tools for safe writes like \"create a sample app\" but gate them behind explicit user confirmation (human-in-the-loop pattern from CopilotKit).\n\n**How to refresh knowledge base automatically?**\nRun explorer agents on a schedule (weekly or on each deploy). Store generation timestamp in the KB header. The prompt cache ensures the refreshed KB is picked up within 5 minutes of the next conversation.\n\n**Multi-language support?**\nClaude natively supports Afrikaans and 100+ languages. Add to the system prompt: \"Respond in the same language the user writes in.\" ElevenLabs supports 70+ languages for TTS.
